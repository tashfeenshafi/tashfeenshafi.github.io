01 // Engineering Research

Tripedal Locomotion: Design Iteration

Most walking robots are over-engineered. They use too many motors, which drives up weight and cost. My research group at KSU wanted to strip that complexity away. Our goal is to build a "tripedal" robot that uses crutch-assisted walking geometry, basically using math instead of motors, to create a cheaper base for robotics.

To get it working, we had to iterate. We ran through several CAD concepts, discussing the pros and cons of each before cutting any metal.

Concept A (My Pitch)
I wanted to minimize the part count to keep it light. It solved the weight issue, but simulations showed it was unstable. A nudge from the front or back would tip it over. We decided not to move forward with this one.
Concept B (Teammate)
Proposed by a teammate to improve the range of motion. It was a solid concept, but the "knee" joint required manufacturing precision that was just too complex for our budget and timeline.
Final Selection (Hybrid)
We compromised. We built a hybrid design that fixed the stability issues of my concept without adding the extra complexity of Concept B.

The Build Phase

With the CAD locked in, we moved to fabrication. We used standard COTS (Commercial Off-The-Shelf) aluminum framing for the build. It wasn't fancy, but it allowed us to test the design immediately without waiting weeks for custom parts.

Right now, I’m working on the control software. The goal is to get this robot climbing stairs and handling obstacles, proving that you don't need expensive hardware to move effectively.

Current physical prototype undergoing testing in the KSU Robotics Lab.

03 // Process Automation

Workflow Optimization

Running a photography business introduced me to a major bottleneck: merging HDR photos. Aligning and merging brackets by hand took me about 5 minutes per set. When I had a batch of 500 photos, that manual work was a massive waste of time.

I wrote a Python script to handle the repetitive tasks. It hooks into the Hugin engine to automatically sort, align, and fuse the images. It cuts my active work time by about 40% and lets me focus on other things while it runs.

# Core logic for batch processing HDR groups def process_groups(groups, output_folder, align_tool, enfuse_tool): if not os.path.exists(output_folder): os.makedirs(output_folder) for group in groups: # 1. Setup paths base_name = os.path.splitext(os.path.basename(group[0]))[0] output_file = os.path.join(output_folder, f"{base_name}_fused.tif") aligned_prefix = os.path.join(output_folder, "temp_aligned_") # 2. Align images using Hugin toolstack subprocess.run([align_tool, "-m", "-a", aligned_prefix] + group) aligned_files = glob.glob(aligned_prefix + "*.tif") # 3. Fuse aligned images into final HDR output cmd = [ enfuse_tool, "--output", output_file, "--exposure-weight=1.0", "--saturation-weight=0.2", "--contrast-weight=0" ] + aligned_files subprocess.run(cmd) # 4. Cleanup temp files for f in aligned_files: os.remove(f)
